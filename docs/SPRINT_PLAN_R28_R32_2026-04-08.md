# Sprint Plan: R28-R32 Demo Readiness

Date: 2026-04-08
Scope: Curated demo-preparation plan after reviewing the repo mission, runbooks, handoffs, sprint docs, validation starter, and the current `scripts/demo.py` behavior.

Companion status board:

## Mission Distilled From The Repo

Across `README.md`, `docs/00_ROADMAP.md`, `docs/HANDOVER.md`, and the design docs, JCoder is trying to be four things at once:

1. A fully local coding assistant that can answer grounded questions about code and technical knowledge.
2. A practical operator tool with reliable CLI and GUI entry points on Windows.
3. A retrieval-backed system that earns trust through evidence, citations, and low-hallucination behavior.
4. A self-improving system that can learn from evals, telemetry, and controlled evolution.

That mission is coherent, but the demo path should not try to prove all four at full depth at the same time.

## What The Demo Actually Needs To Prove

The current docs imply a "demo" means more than a health-check tour. A real JCoder demo should prove:

1. The operator can launch the system reliably from the documented path.
2. JCoder can ingest or load a known repo/index and answer a grounded code question with evidence.
3. JCoder can complete one bounded coding-assistant task end to end, including verification.
4. JCoder fails safely on unanswerable or adversarial prompts instead of bluffing.
5. The GUI and CLI tell the same story, even if the CLI remains the primary proof path.

Anything beyond that, including evolution, distillation, or multi-agent research-lab behavior, is a stretch objective and should be framed as such.

## Current Reality After Reviewing Docs + Live Demo

### What is strong already

- The mission is well documented and consistently points toward a local, grounded coding assistant.
- The repo has substantial retrieval, agent, GUI, and self-learning infrastructure.
- The GUI command-center path now has parity and smoke coverage.
- The validation and recovery docs are directionally correct: they insist on trusted corpus -> reindex -> query validation -> regression -> demo.

### What is not demo-ready yet

- The current `scripts/demo.py` is mostly a system walkthrough, not a coding-assistant proof.
- The current demo does not exercise `ingest`, `ask`, agent task execution, code editing, or test-running as a single narrative.
- The current demo output includes warning noise and index-discovery double counting when `JCODER_DATA` falls back to `data/`.
- The docs disagree on important live numbers such as test counts, index counts, and what is "canonical", which weakens operator trust.
- Existing long-range sprint plans mix real near-term demo work with far-future autonomy/evolution goals.

## Planning Rules For The Next Sprints

These rules are taken seriously because the repo history shows repeated optimism drift:

1. No retuning on top of untrusted source data or unclear index state.
2. No "demo ready" claim without an explicit pass/fail checklist and fresh independent QA.
3. No operator demo that depends on hidden environment knowledge.
4. No demo script that only proves health checks while implying product proof.
5. Self-learning and research-lab lanes stay secondary until the basic demo story is repeatable.

## Sprint R28: Demo Contract And Canonical Truth

Priority: P0

Goal: Replace "demo as vague walkthrough" with a single canonical contract and current-state baseline.

### Tasks

- R28.1: Audit doc drift across `README.md`, `docs/OPERATIONAL_RUNBOOK.md`, `docs/HANDOVER.md`, sprint status docs, and validation docs.

- One deterministic demo corpus exists.

## Sprint R29: Retrieval Trust And Safety Gate

Priority: P0

Goal: Earn the right to demo by proving the retrieval and safety substrate is trustworthy.

### Tasks

- R29.1: Make trusted-source status, corpus integrity status, reindex status, and query-validation status visible in one place.
- R29.2: Run the validation starter flow on a small golden corpus and record parser/retrieval behavior.
- R29.3: Enforce a demo answer format that distinguishes answer, evidence, derived inference, confidence, and missing data.
- R29.4: Add explicit demo-lane checks for unanswerable prompts and prompt-injection resistance.
- R29.5: Require a fresh independent QA pass before declaring the retrieval lane demo-ready.

### Exit Criteria

- Retrieval lane passes a deterministic golden corpus check.
- Safety lane passes unanswerable and injection cases.
- Independent QA signoff is explicit.
- Retuning approval is explicit instead of implied.

## Sprint R30: End-To-End Coding Assistant Proof

Priority: P0

Goal: Turn the demo from a "search tour" into a bounded coding-assistant proof.

### Tasks

- R30.1: Define two demo scenarios:
  - repo comprehension: ingest/load -> ask -> grounded answer with evidence
  - bounded implementation: task -> code change -> tests -> summarized outcome
- R30.2: Build a scripted CLI demo harness that runs those scenarios end to end on a deterministic sample repo.
- R30.3: Capture proof artifacts: command transcript, diff summary, test result, and rollback/reset instructions.
- R30.4: Separate mock-mode proof from live-stack proof so each has honest expectations.
- R30.5: Add regression coverage for the demo harness itself.

### Exit Criteria

- A fresh operator can run one bounded CLI coding demo from documented steps.
- The demo produces proof artifacts, not just terminal chatter.
- The mock path is deterministic and the live path is clearly labeled.

## Sprint R31: GUI And Operator Hardening

Priority: P1

Goal: Make the operator experience low-friction and consistent with the CLI proof path.

### Tasks

- R31.1: Align `gui_launch.bat`, `python main.py gui`, `jcoder-gui`, and the runbook into one canonical launch story.
- R31.2: Define a GUI demo flow that mirrors the CLI story: doctor, load config, ingest sample, ask sample, inspect output.
- R31.3: Expand GUI button-smash/parity/regression coverage around the actual demo path, not only around generic command wiring.
- R31.4: Remove or quarantine operator-confusing warnings, stale references, and duplicate instructions from runbooks.
- R31.5: Run a crash/recovery drill so the operator can recover from a failed demo run without improvisation.

### Exit Criteria

- GUI demo path is documented and smoke-tested.
- Launcher behavior is unsurprising on Windows.
- Operator recovery steps are explicit and current.

## Sprint R32: Live-Stack Credibility And Stretch Story

Priority: P1

Goal: Prepare the post-demo narrative for the real local-model stack without blocking the core demo.

### Tasks

- R32.1: Freeze the intended live-stack environment and GPU assignment story.
- R32.2: Run a live-stack version of the bounded demo on the approved local model path.
- R32.3: Record latency, answer quality, retrieval behavior, and failure behavior for the live stack.
- R32.4: Decide which self-learning or distillation claims are ready for demo mention and which remain "next" work.
- R32.5: Publish a clear split between "demoed now" and "roadmap after demo".

### Exit Criteria

- A live-stack evidence pack exists.
- GPU assignment and backend assumptions are documented.
- Stretch features are described honestly and separately from the core proof.

## Sprint R33: Post-Demo Learning Loop Proof

Priority: P2

Goal: Only after the core demo is stable, prove that JCoder can improve itself in a measurable and trustworthy way.

### Tasks

- R33.1: Freeze a clean baseline on the bounded demo tasks and eval slices.
- R33.2: Run one controlled learning or distillation cycle with immutable before/after artifacts.
- R33.3: Require external verification and rollback discipline per `docs/FLYWHEEL_TEST_PROTOCOL.md`.
- R33.4: Show whether the system improved on held-out tasks, not just seen prompts.

### Exit Criteria

- Self-improvement claims are evidence-backed.
- Rollback and contamination controls are demonstrated.

## Recommended Execution Order

R28 -> R29 -> R30 -> R31 -> R32

R33 stays explicitly post-demo unless R30-R32 finish early and remain green.

## What Should Not Block The Demo

- VM tournament mode
- recursive meta-learning
- autonomous research-lab claims
- broad multi-agent mesh ambitions
- cloud burst infrastructure

Those are still valid roadmap items. They are not the next operator-facing proof.

## Definition Of Demo Readiness

JCoder is demo-ready only when all of the following are true:

1. The operator can launch the documented CLI and GUI paths without guessing.
2. The corpus and index state used for the demo is explicit and trusted.
3. The system answers at least one grounded repo question with evidence.
4. The system completes at least one bounded coding-assistant task with verification.
5. The system handles at least one unsafe or unanswerable case correctly.
6. A fresh independent QA pass confirms the above.

If any of those are missing, the repo may be promising, but it is not demo-ready.
